# adreg
A project to register users via a registration link directly into Active Directory. Allows an admin to invite and add AD users via a registration link from the web app.

## Project structure

The project consists of two components:
1. **Admin App**: Allows an administrator to generate an invitation link and send it to a user by email. An administrator can approve or reject users who registered via a link. **This app should only be available on the internal network and must not be exposed to the internet.**
2. **Registration App**: A user can register themselves via a link generated by the admin app. This app is designed to be exposed to the internet.

## Requirements

- A computer, VM or server that can run Docker.
- Docker and Docker Compose set up.
- A reverse proxy to provide TLS for the apps. The apps do not manage TLS certificates themselves and are designed to run behind a reverse proxy.
- Firewall rules to allow the registration app to be available from outside the network if you want users to register to your AD via the Internet.

## Setup

Download the `compose.yml` and the `.env` file, configure the variables inside the `.env` file and run `docker compose up -d` to start.

### Step 0: Download `compose.yml` and `.env` to a host folder.

```
curl -O https://raw.githubusercontent.com/we-kode/adreg/master/compose.yml
curl -O https://raw.githubusercontent.com/we-kode/adreg/master/.env
```

### Step 1: Configure Active Directory (LDAP / AD)

This project integrates directly with Active Directory / LDAP. The Admin app will use the configured LDAP connection to create users and add them to groups.

Important notes:

- Setting `unicodePwd` (password) typically requires LDAPS (TLS, port 636) and a certificate trusted by the client.
- The bind account (Service Account) must have permission to create users in the target OU and to modify group membership (`member` attribute).
- `SearchBase` is the base DN for searches (e.g. `DC=example,DC=local`).
- `UsersContainer` can be an OU relative to `SearchBase` (e.g. `OU=Users`) or a full DN (e.g. `OU=Users,DC=example,DC=local`).

Example `.env` configuration for Windows Active Directory (replace with your values):

```
# Active Directory / LDAP settings (example for Windows AD with LDAPS)
AD__LdapUrl=ldaps://dc01.example.local:636
AD__BindDn=CN=adregsvc,OU=Service Accounts,DC=example,DC=local
AD__BindPassword=VerySecretPassword123!
AD__SearchBase=DC=example,DC=local
AD__UsersContainer=OU=Users
```

Fields explained:

- `AD__LdapUrl`: LDAPS URL or host:port. Example: `ldaps://dc01.example.local:636` or `ad.example.local:389` (LDAP, insecure).
- `AD__BindDn`: Distinguished Name of the service account, e.g., `CN=adregsvc,OU=Service Accounts,DC=example,DC=local`.
- `AD__BindPassword`: Password of the service account.
- `AD__SearchBase`: Base DN for search (domain root), e.g., `DC=example,DC=local`.
- `AD__UsersContainer`: OU or container for new users (relative to `SearchBase` or absolute DN).

PowerShell examples (on a computer with RSAT / access to AD):

- Determine the base DN of the domain:
```
Get-ADDomain | Select-Object DistinguishedName
```
- DN of a user / service account:
```
Get-ADUser -Identity adregsvc | Select-Object DistinguishedName
```
- List of groups (Name + DN):
```
Get-ADGroup -Filter * | Select-Object Name, DistinguishedName
```

User permissions (minimum):

- On the target OU: `Create Child Objects` (or Create User).
- On groups: `Write Members` permission or the ability to modify the `member` attribute.
- If passwords are set via LDAPS, LDAPS must be correctly configured on the DC (certificate, etc.).

LDAPS / Certificates:

- For `ldaps://`, the domain controller requires a certificate trusted by the client (e.g., from an internal CA).
- In test environments, a self-signed certificate is possible, but the client must configure this certificate as trusted.

How `ADService` accepts group values:

- Accepted values: the group’s full DN (e.g., `CN=IT-Helpdesk,OU=Groups,DC=example,DC=local`) or just the `cn` value (e.g., `IT-Helpdesk`).
- If there is no `=` in the group specification, `ADService` searches by `cn`.

Error handling / Fallback:

- If `unicodePwd` cannot be set (no LDAPS), the account is created anyway; setting the password fails and is logged. Set the password manually or via script.

Quick checklist before deployment:
1. Test LDAPS on the DC (e.g., `openssl s_client -connect dc01.example.local:636`).
2. Verify the Bind account DN and password.
3. Check `SearchBase` (domain DN).
4. `UsersContainer` exists, and the Bind account can create users there.
5. Groups in AD are visible and searchable via `cn`.

### Step 2: Setup the `.env`
1. Configure all required variables for the mail SMTP server and Active Directory (see example above).
2. Optional and default values can be adjusted as needed.
3. Start the apps. The admin app runs by default on port 8081 and the registration app on port 8082.
